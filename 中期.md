## 一、课题回顾

### 1.1 研究背景

随着云原生技术的普及，微服务架构已成为主流。然而，微服务系统故障排查面临三大挑战：

- **故障定位难**：海量指标、日志、追踪数据，人工排查效率低
- **知识传承难**：专家经验难以固化和复用，新人上手周期长
- **响应速度慢**：故障期间每秒钟都造成损失，急需自动化辅助

### 1.2 研究目标

本课题设计并实现一个**基于 SOP（标准操作流程）的云原生微服务故障根因分析系统**，核心能力：

1. **自动化诊断**：接收故障告警，自动执行诊断流程
2. **知识复用**：基于历史故障和 SOP 知识库辅助决策
3. **多轮收敛**：一次诊断未能定位时，通过问题细化实现迭代收敛
4. **可解释输出**：给出根因结论、证据链和修复建议

### 1.3 技术选型

| 组件 | 技术方案 | 说明 |
|------|----------|------|
| 编排框架 | LangGraph | 基于状态机的 Agent 协作框架 |
| LLM | DeepSeek API | 根因判定、动作选择、代码生成 |
| 知识库 | ChromaDB（向量库）+ SQLite | SOP 存储、历史故障匹配 |
| 运行环境 | K3s 集群（虚拟机） | 真实微服务环境 |
| 监控数据 | kubectl / metrics-server | Pod CPU、内存、事件等 |

### 1.4 论文参考

本系统参考论文 **"Flow-of-Action SOP"** 的四场景设计：

| 场景 | 描述 | 流程 |
|------|------|------|
| **正常流程** | 匹配到 SOP，一次找到根因 | match_sop → run_sop → ob_agent → judge_agent → 终止 |
| **收敛机制（情况二）** | 未找到根因，生成细化问题回流 | judge → 生成 refined_problem → match_sop 重新匹配 |
| **无 SOP 匹配** | 知识库无匹配 SOP，自动生成 | match_sop 失败 → generate_sop → 继续 |
| **底层查询** | 专家级深入查询 | ob_agent / judge_agent 可触发工具调用 |

**核心流程说明**：

1. **match_sop**：接收故障描述，向量检索匹配 SOP 知识库
2. **action_agent**：基于当前状态和 SOP，生成候选动作集（5个候选）
3. **main_agent**：LLM 决策，选择最优动作
4. **code_agent**：如需代码生成，SOP → Python 代码
5. **tool_executor**：执行工具（kubectl / metrics / logs）
6. **ob_agent**：分析观测数据，提取故障线索
7. **judge_agent**：判定根因，支持多轮收敛
8. **generate_report**：输出根因报告

### 2.3 状态机设计（结合代码讲解）

LangGraph 状态机是系统的核心，运行流程如下：

#### 2.3.1 状态定义（state.py）

```python
class RCAState(TypedDict):
    fault_info: str                          # 故障信息
    matched_sop: Optional[Dict]              # 匹配的SOP
    candidate_action_set: Optional[List]    # 候选动作集
    selected_action: Optional[Dict]          # 选中的动作
    current_observation: Optional[str]       # 观测结果
    generated_code: Optional[str]            # 生成的代码
    extracted_clues: Optional[Dict]          # 提取的线索
    judge_result: Optional[Dict]             # 根因判定结果
    executed_steps: List[Dict]               # 已执行步骤
    iteration_count: int                     # 循环次数
    consecutive_no_gain: int                 # 连续无增益次数
    refined_problem_statement: Optional[str] # 细化问题描述（收敛用）
    should_terminate: bool                  # 是否终止
```

#### 2.3.2 图构建（build_graph.py）

```python
def create_rca_graph():
    graph = StateGraph(RCAState)

    # 添加节点
    graph.add_node("match_sop", match_sop_node)
    graph.add_node("action_agent", action_agent_node)
    graph.add_node("main_agent", main_agent_node)
    graph.add_node("code_agent", code_agent_node)
    graph.add_node("tool_executor", tool_executor_node)
    graph.add_node("ob_agent", ob_agent_node)
    graph.add_node("judge_agent", judge_agent_node)
    graph.add_node("generate_report", generate_report_node)

    # 条件边1：MainAgent 根据选中动作分流
    graph.add_conditional_edges(
        "main_agent",
        route_after_main_agent,
        {
            "code_agent": "code_agent",          # 生成代码
            "generate_sop": "generate_sop",       # 生成新SOP
            "tool_executor": "tool_executor",     # 执行工具
            "match_sop": "match_sop",            # 重新匹配SOP
            "match_observation": "match_observation",  # 匹配历史故障
            "generate_report": "generate_report"   # 生成报告
        }
    )

    # 条件边2：JudgeAgent 根据判定结果分流
    graph.add_conditional_edges(
        "judge_agent",
        route_after_judge_agent,
        {
            "action_agent": "action_agent",      # 继续诊断
            "generate_report": "generate_report", # 找到根因，终止
            "match_sop": "match_sop"             # 收敛回流
        }
    )

    return graph.compile()
```

**完整的状态流转**：

```
match_sop ──→ action_agent ──→ main_agent ──┬──→ code_agent ──→ action_agent
                                            ├──→ generate_sop ──→ action_agent
                                            ├──→ tool_executor ──→ action_agent ──→ ...
                                            ├──→ match_sop ──→ action_agent
                                            ├──→ match_observation ──→ ob_agent ──→ judge_agent
                                            └──→ generate_report (终止)

其中 ob_agent ──→ judge_agent 后：
    judge_agent ──┬──→ generate_report (找到根因，终止)
                   ├──→ match_sop (收敛回流)
                   └──→ action_agent (继续诊断)
```

#### 2.3.3 节点实现（nodes.py）

以 **judge_agent_node** 为例，这是收敛机制的核心：

```python
def judge_agent_node(state: RCAState) -> Dict[str, Any]:
    # 调用 LLM 判定是否找到根因
    judge_result = judge_root_cause(...)

    is_root_cause_found = judge_result.get("is_root_cause_found", False)

    # 关键：收敛机制实现
    if not should_terminate:
        if not is_root_cause_found:
            # 未找到根因，无增益次数 +1
            new_no_gain = consecutive_no_gain + 1

            # 收敛触发：生成细化问题描述
            if new_no_gain >= 1 and extracted_clues:
                refined_problem_statement = generate_refined_problem(
                    fault_info, extracted_clues, executed_steps
                )
        else:
            new_no_gain = 0  # 找到根因，重置计数

    return {
        "judge_result": judge_result,
        "consecutive_no_gain": new_no_gain,
        "refined_problem_statement": refined_problem_statement
    }
```

#### 2.3.4 路由函数

**route_after_main_agent**：根据 MainAgent 选中的动作决定下一步

```python
def route_after_main_agent(state: RCAState) -> str:
    """MainAgent之后的路由"""
    selected = state.get("selected_action", {})
    action = selected.get("action", "")

    if action == "generate_code":
        return "code_agent"
    elif action == "generate_sop":
        return "generate_sop"
    elif action == "match_sop":
        return "match_sop"
    elif action == "match_observation":
        return "match_observation"
    elif action == "generate_report":
        return "generate_report"
    else:
        return "tool_executor"
```

**route_after_judge_agent**：根据 JudgeAgent 判定结果决定下一步

```python
def route_after_judge_agent(state: RCAState) -> str:
    should_terminate = state.get("should_terminate", False)
    judge_result = state.get("judge_result", {})
    is_root_cause_found = judge_result.get("is_root_cause_found", False) if judge_result else False
    refined_problem = state.get("refined_problem_statement")

    if should_terminate or is_root_cause_found:
        return "generate_report"

    if refined_problem:
        return "match_sop"

    return "action_agent"
```

### 2.4 四大 Agent 详解

| Agent | 输入 | 输出 | 核心能力 |
|-------|------|------|----------|
| **ActionAgent** | 故障信息 + SOP + 已执行步骤 | 5个候选动作及其解释 | 基于规则的候选生成 |
| **MainAgent** | 候选动作集 + 上下文 | 选中的最优动作 | LLM 决策（DeepSeek） |
| **ObAgent** | 观测数据 + 历史故障 | 故障类型、位置、线索、可能根因、排除根因 | LLM 分析提取 |
| **JudgeAgent** | 线索 + 执行历史 | 是否找到根因、证据链、修复建议 | LLM 判定 + 防死循环 |

### 2.5 知识库设计

#### 2.5.1 SOP 知识库（向量检索）

- **向量检索**：将故障描述向量化，在 ChromaDB 中检索最相似的 SOP
- **已有 SOP**：CPU过高诊断SOP、网络故障诊断SOP 等

#### 2.5.2 历史故障知识库

- **用途**：match_observation 步骤，检索相似历史故障
- **字段**：fault_id, fault_info, fault_type, observation, root_cause, similarity_score

### 2.6 系统效果演示（真实环境测试）

以 **"productcatalogservice CPU过高"** 为输入，在 K3s 集群真实运行：

```
[1]  match_sop     → 匹配到"CPU过高诊断SOP"（距离0.233，匹配度高）
[2]  action_agent  → 生成4个候选动作
[3]  main_agent    → 选择 generate_code
[4]  code_agent    → 生成 SOP 执行代码（399字符）
[5]  action_agent  → 生成5个候选动作
[6]  main_agent    → 选择 run_sop
[7]  tool_executor → 执行 SOP：获取CPU指标 + Pod状态
[8]  action_agent  → 生成2个候选
[9]  main_agent    → 选择 match_observation
[10] match_observation → 找到1个相似历史故障（CPU limit配置过低）
[11] ob_agent      → 提取线索：CPU 250m，limit 250m，request 20m，重启21次
[12] judge_agent   → 判定根因：CPU资源配置不合理（request远低于limit）
[13] generate_report → 输出根因报告
```

**根因判定结果**：

> **最终根因**：productcatalogservice的CPU资源配置不合理：CPU request（20m）设置过低，导致调度不足，Pod在负载下容易达到CPU limit（250m）并引发节流，进而导致高使用率和频繁重启。

**输出报告包含**：
- 故障信息与匹配 SOP
- 诊断路径（APL = 2）
- 历史相似故障参考
- ObAgent 提取的5条关键线索
- JudgeAgent 根因判定及证据链
- 3条修复建议
